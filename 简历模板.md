## 🔑 面试追问预判（提前准备这些回答）

| 面试官可能会问 | 参考回答要点 |
|--------------|------------|
| **"为什么用 LLM 做测试用例生成？"** | 传统手工写用例效率低且依赖个人经验，LLM 能理解自然语言需求、批量生成结构化用例，结合 8 种设计方法保证覆盖度。本质是用 AI 放大测试工程师的能力。 |
| **"LLM 输出不稳定怎么解决？"** | 4 层容错流水线：①正则提取 JSON 数组 → ②让 LLM 自我修正格式 → ③移除尾部逗号/BOM/中文引号 → ④正则逐条提取 JSON 对象作为兜底。每层独立，逐级降级。 |
| **"Prompt 怎么设计的？"** | 双层架构。System Prompt 定义角色（10 年测试架构师）、输出格式（10 字段 JSON）、优先级标准。User Prompt 注入具体需求 + 选定的设计方法 + 模块数要求。通过约束输出格式来保证可解析。 |
| **"覆盖度评分模型怎么设计的？"** | 4 维度各 25 分：模块覆盖（模块数×8）、类型覆盖（类型数×8）、方法覆盖（方法数×8）、优先级分布合理性（P0 10-25% + P1 25-45% 各 12.5 分）。≥80 分优秀，60-79 良好，<60 需优化。 |
| **"这个项目还有什么改进空间？"** | ①接入 RAG 知识库复用历史用例 ②多轮对话式迭代优化 ③对接禅道/Jira API 一键提用例 ④增加测试脚本自动生成（Playwright/Pytest）⑤支持批量需求文档导入。 |
| **"遇到的最大技术难点？"** | LLM JSON 输出稳定性问题。最初的纯正则提取成功率约 70%，通过加入 LLM 自修复、兼容性清洗和兜底策略，将成功率提升到接近 100%。这个过程中学到了工程化容错设计的重要性。 |
